# डेटाबेस

<div class="article-intro">

ChurchApps API एक **मॉड्यूल-प्रति-डेटाबेस** आर्किटेक्चर का उपयोग करता है। छः डेटा मॉड्यूलों में से प्रत्येक के अपने MySQL डेटाबेस हैं एक स्वतंत्र कनेक्शन पूल के साथ, स्पष्ट डेटा सीमाएं प्रदान करते हुए सबकुछ एक ही परिनियोजन के भीतर रखते हैं।

</div>

<div class="prereqs">
<h4>शुरू करने से पहले</h4>

- **MySQL 8.0+** स्थापित करें -- [पूर्वापेक्षाएं](../setup/prerequisites) देखें
- अपनी `.env` फ़ाइल में डेटाबेस कनेक्शन स्ट्रिंग कॉन्फ़िगर करें -- [पर्यावरण चर](../setup/environment-variables) देखें

</div>

## आर्किटेक्चर अवलोकन

```
Api
├── membership_db   ← लोग, समूह, अनुमतियां
├── attendance_db   ← सेवाएं, सत्र, रिकॉर्ड
├── content_db      ← पृष्ठ, अनुभाग, तत्व
├── giving_db       ← दान, निधि, भुगतान
├── messaging_db    ← बातचीत, सूचनाएं
└── doing_db        ← कार्य, योजनाएं, असाइनमेंट
```

### मुख्य डिज़ाइन निर्णय

- **प्रति मॉड्यूल एक डेटाबेस** -- प्रत्येक मॉड्यूल अपने MySQL डेटाबेस को एक समर्पित कनेक्शन पूल (`KyselyPool` द्वारा प्रबंधित) के साथ बनाए रखता है। यह मॉड्यूल को अलग रखता है और स्वतंत्र स्कीमा विकास की अनुमति देता है।
- **एक्सक्लूसिव स्वामित्व** -- एक मॉड्यूल की तालिकाएं केवल उस मॉड्यूल के अपने कोड द्वारा पढ़ी और लिखी जाती हैं। जब किसी अन्य मॉड्यूल को डेटा की आवश्यकता होती है, तो यह तालिकाओं को स्वयं क्वेरी करने के बजाय मालिकाना मॉड्यूल के गेटवे को कॉल करता है -- [क्रॉस-मॉड्यूल संचार](./module-structure#cross-module-communication) देखें।
- **ORM के बिना रिपोजिटरी पैटर्न** -- सभी डेटा एक्सेस रिपोजिटरी वर्गों के माध्यम से जाता है जो मॉड्यूल की स्कीमा के विरुद्ध Kysely क्वेरी बिल्डर के साथ टाइप किए गए SQL बनाते हैं। यह क्वेरी प्रदर्शन और व्यवहार पर पूर्ण नियंत्रण देता है।
- **डिजाइन द्वारा बहु-किरायेदार** -- हर क्वेरी `churchId` द्वारा कवर की जाती है। सभी तालिकाओं में एक `churchId` स्तंभ शामिल है, और रिपोजिटरी परत स्वचालित रूप से किरायेदार अलगाव लागू करता है।

## कनेक्शन स्ट्रिंग

प्रत्येक मॉड्यूल का डेटाबेस कनेक्शन `.env` में मानक MySQL कनेक्शन स्ट्रिंग प्रारूप का उपयोग करके कॉन्फ़िगर किया गया है:

```
mysql://user:password@host:port/database
```

उदाहरण के लिए, स्थानीय विकास सेटअप इस तरह दिख सकता है:

प्रत्येक मॉड्यूल अपने कनेक्शन को `<MODULE>_CONNECTION_STRING` नामित एक पर्यावरण चर से पढ़ता है:

```env
MEMBERSHIP_CONNECTION_STRING=mysql://root:password@localhost:3306/churchapps_membership
ATTENDANCE_CONNECTION_STRING=mysql://root:password@localhost:3306/churchapps_attendance
CONTENT_CONNECTION_STRING=mysql://root:password@localhost:3306/churchapps_content
GIVING_CONNECTION_STRING=mysql://root:password@localhost:3306/churchapps_giving
MESSAGING_CONNECTION_STRING=mysql://root:password@localhost:3306/churchapps_messaging
DOING_CONNECTION_STRING=mysql://root:password@localhost:3306/churchapps_doing
```

:::info
उत्पादन में, कनेक्शन स्ट्रिंग AWS SSM पैरामीटर स्टोर में संग्रहीत होती हैं और स्टार्टअप पर `Environment` वर्ग द्वारा पढ़ी जाती हैं।
:::

## स्कीमा स्क्रिप्ट

तालिका स्कीमा \`tools/migrations/`\ निर्देशिका में Kysely माइग्रेशन के रूप में परिभाषित किए गए हैं, मॉड्यूल द्वारा व्यवस्थित:

```
tools/migrations/
├── membership/
├── attendance/
├── content/
├── giving/
├── messaging/
└── doing/
```

माइग्रेशन तालिका निर्माण, सूचकांक, और स्कीमा परिवर्तन को परिभाषित करते हैं। \`tools/dbScripts/`\ निर्देशिका में डेमो और बीज डेटा होता है जो स्कीमा के शीर्ष पर लोड किया जा सकता है।

## डेटाबेस प्रारंभीकरण

### सभी डेटाबेस को प्रारंभ करें

```bash
npm run initdb
```

यह सभी छः डेटाबेस बनाता है और प्रत्येक के लिए माइग्रेशन चलाता है।

### एक ही मॉड्यूल को प्रारंभ करें

```bash
npm run initdb -- --module=membership
```

:::tip
किसी विशिष्ट मॉड्यूल पर काम करते समय, आप केवल उस मॉड्यूल के डेटाबेस को पुनः प्रारंभ कर सकते हैं अन्य को प्रभावित किए बिना।
:::
