# Журнал аудита и отменяемые партии

<div class="article-intro">

Каждая инициированная пользователем мутация в API записывается — кто, что, когда и откуда — во всех модулях без какой-либо разводки контроллера. Поверх этого реестра лежит слой партии: импорт или массовое действие может быть помечено как партия и позже **отменено** по строкам, в стиле Planning Center. Оба находятся в единственной таблице `auditLogs` в базе данных членства и полностью управляются с одной точки отсечки, `BaseController.actionWrapper`. На этой странице описано, что аудируется, где живут данные, компромиссы производительности, которые их формируют, и как отмена обращает вспять партию безопасно без кросс-базовых транзакций.

</div>

## Обзор

```
каждый запрос мутирующего (POST/PUT/PATCH/DELETE)
        │
        ▼
BaseController.actionWrapper ──▶ вывести {module, entityType, category, action}
        │                         из req.baseUrl + метода  (AUDIT_REGISTRY = переопределения/исключения только)
        │
        ├─ нормальный режим ────▶ выполнить действие ─▶ await AuditLogHelper.log(after-values)  ──┐
        │                                        (удаления также захватывают before-image)     │
        │                                                                                  ▼
        └─ X-Batch-Id присутствует ──▶ снимок before-images (strict) ─▶ выполнить действие ─▶ строки аудита помечены batchId
                                                                                           │
                                                                                           ▼
                                                             auditLogs  (membership DB, одна таблица, все модули)
                                                                                           │
   POST /membership/batches/:id/undo ──▶ BatchUndoHelper ──▶ пройти строки вспять, per entity ┘
                                          conflict guard → восстановить / удалить / пересоздать
```

Два структурных факта управляют всем ниже:

1. **Слой контроллера — единственное место, где знает актер.** Репозитории никогда не видят `AuthenticatedUser`; только контроллеры хранят `au`. Контроллеры каждого модуля уже проходят через `BaseController.actionWrapper`, поэтому здесь зацепляется аудирование — нет изменений подписей репо.
2. **Одна таблица служит всем модулям.** Строки аудита для пожертвований, посещаемости, контента и т.д. все записываются в `auditLogs` DB членства через `RepoManager.getRepos("membership")`, даже из контроллера не-членства. "Всё, что Джейн изменила сегодня" остается одним запросом.

## Что аудируется

Аудирование **включено по умолчанию для каждого мутирующего глагола на каждом маршруте**. `actionWrapper` выводит поля аудита из запроса без конфигурации на маршрут:

| Поле | Выведено из |
|-------|--------------|
| `module` | `this.moduleName` (владеющий модуль) |
| `entityType` | сингуляризованный последний сегмент `req.baseUrl` (например, `/membership/people` → `person`) |
| `category` | по умолчанию `entityType` |
| `action` | `${entityType}_saved` для `POST /`, `${entityType}_deleted` для `DELETE /:id`, иначе `${entityType}_${method}:${routePath}` поэтому не-CRUD маршруты (например, `task_post:/:id/move`) захватываются автоматически |

`BaseController.AUDIT_REGISTRY` **только для переопределений и исключений** — это не список разрешений. Маршрут появляется там для переименования его категории/типа сущности, для объявления `{ dbModule, table }` (что делает его способным для партии и отмены), для его маркировки как `sensitive` (аудит анонимных мутаций), или для отключения с помощью `optOut: true`.

**Список исключений** (пути записи пожарного потока, которые будут тонуть в реестре): посещаемость `visits` / `visitsessions` / `sessions` / `checkin` (воскресная буря check-in) и обмен сообщениями `messages` / `connections` / `devices` (чат и присутствие). Всё остальное логируется.

**Массовые конечные точки** (`people/bulk-delete`, `people/bulk-update`, `groupmembers/bulk-add`, `groupmembers/bulk-remove`) зарегистрированы в `BULK_ROUTES` и выдают **одну строку аудита на затронутый id**, поэтому импорт 10k человек производит 10k строк — эта детализированная сущность и делает партию отменяемой.

**Анонимные мутации** (`actionWrapperAnon` — пожертвование гостя, регистрация гостя, отправка формы) аудируются только для маршрутов с пометкой реестра `sensitive`, записываются с `userId="anonymous"` плюс IP клиента. Пожертвования ведут список; этот путь имеет реальную историю регрессии.

### Редакция секретов и ограничения размера

Перед хранением любого полезного нагруженного `details`, `AuditLogHelper.capDetails()` выполняет `sanitizeValue()` над ним:

- **Секретные ключи редактируются.** Любое поле, имя которого в нижнем регистре находится в `SENSITIVE_KEYS` (`password`, `token`, `cvv`, `cardnumber`, `routing_number`, `accesstoken`, `clientsecret`, …) заменяется на `"[redacted]"`.
- **Огромные скаляры удаляются.** Любой `data:` URI или строка более 4 KB (фото в base64, blobs) становится `"[stripped]"`.
- **Переразмерные строки обрезаны.** Если сериализованный JSON превышает ~64 KB, весь полезный нагруженный заменяется на `{ truncated: true }`. Обрезанные строки все еще просматриваются — но **не отменяемы** (нет перед/после образа для восстановления).

## Где живут данные

Единственная таблица `auditLogs` в базе данных **membership** поддерживает каждый модуль. Столбцы: `id, churchId, userId, category, action, entityType, entityId, details (MEDIUMTEXT JSON string), ipAddress, module, batchId, created`. Миграция `tools/migrations/membership/2026-07-04_audit_universal.ts` добавляет `module` + `batchId`, расширяет `details` с `TEXT` на `MEDIUMTEXT`, добавляет индексы `ix_auditLogs_batch (batchId)` и `ix_auditLogs_entity (churchId, module, entityType, entityId, created)` и создает таблицу `batches`. Столбец `module` существует именно для того, чтобы столкновения `entityType` в разных модулях (`note`, `setting` существуют в нескольких) оставались фильтруемыми, и индекс сущности — это то, что питает как историю per-entity, так и защиту конфликтов отмены.

Кросс-модульные записи идут через `RepoManager.getRepos("membership")` изнутри обертки. Упорядочение намеренное: **основная запись фиксируется в модуле DB сначала, вставка аудита второй.** В нормальном режиме отказ от вставки аудита проглатывается (`console.error`, Sentry подхватывает это) — аудит рекомендательный и должен никогда не срывать запрос пользователя. В **режиме партии это строгое** (см. ниже).

:::info Почему не триггеры, CDC или таблицы per-модули?
- **MySQL триггеры** не знают действующего пользователя (подключение не имеет `au`), и будут означать поддержку наборов триггеров в каждой схеме.
- **binlog / CDC** — это весь проект инфраструктуры с той же проблемой идентификации актера.
- **Резьба `userId` через каждый репо** будет означать трогать сотни файлов для перемещения информации, которую слой контроллера уже имеет.
- **Таблицы аудита Per-модули** означали бы 7× больше сантехники и запросы fan-out для любого вопроса кросс-модуля. Одна таблица в точке отсечения контроллера — это дизайн с наименьшим кодом, который все еще захватывает актера.
:::

## Позиция производительности

Горячий путь намеренно дешевый; стоимость платится только где это что-то покупает.

- **Нет чтения-перед-записью в нормальных обновлениях.** Обычное сохранение **не** загружает старую запись. **Отправленные after-values** хранятся в `details.after`; UI реконструирует old→new в *view* время путем диффирования против предыдущей записи аудита сущности. Один запрос в view time, нулевая стоимость в write time. Поля, никогда не затронутые с запуска, просто не показывают значение "old" — приемлемо.
- **Удаления получают before-image.** `DELETE /:id` на маршруте реестра с `{ dbModule, table }` загружает строку обобщенно сначала и хранит её в `details.before`. Удаления редки и before-image — это вся судебная стоимость.
- **Режим партии — единственное систематическое чтение-перед-записью**, и это opt-in — операция bulk/import уже дорогая, поэтому N snapshotные чтения — цена отмены.
- **Вставки аудита ожидаются.** `actionWrapper` собирает обещания логирования и `await Promise.allSettled(...)` перед возвратом. Это единственный самый важный инвариант: на Lambda контейнер **замораживается в мгновение возврата ответа**, поэтому неожиданная вставка молча опускается. "Fire and forget" здесь означает *ошибки никогда не срывают запрос*, не *не ожидайте* — одна вставка на уже теплый пул членства ~1–3 ms.

## Партии и отмена

**Партия** группирует набор мутаций так, чтобы они могли быть просмотрены и перепроверены вместе. Есть два способа открыть одну:

- **Явный:** `POST /membership/batches { label, source }` возвращает `batchId`. Клиент (B1Transfer, B1Admin import UI) затем отправляет `X-Batch-Id: <id>` на каждое последующее сохранение/удаление. `POST /membership/batches/:id/complete` закрывает его и штампует `itemCount`.
- **Неявный:** четыре массовые конечные точки открывают, заполняют и завершают свою собственную партию внутри одного запроса, возвращая `batchId` в ответе.

Таблица `batches` (membership DB): `id, churchId, userId, label, source, status (open|completed|undone|partial|failed), itemCount, created, completedAt, undoneAt`.

### Режим партии строгий

Когда `X-Batch-Id` присутствует, `actionWrapper` затягивает каждую защиту (`writeBatchAuditRows`):

1. Партия должна существовать, быть `open` и принадлежать `au.churchId` — иначе **403**.
2. Маршрут должен быть способен к партии (`{ dbModule, table }` в реестре) — иначе **400**.
3. Перед запуском действия, before-images для всех затронутых ids загружены в **одном** `WHERE id IN (...) AND churchId = ?` запросе. Если это snapshotное чтение срывается, запрос **срывается 500 и действие не выполняется** — режим партии должен никогда не производить необратимый реестр молча. (Нормальный режим, напротив, best-effort и проглатывает snapshot отказы.)
4. После успеха действия, одна строка аудита на сущность написана с `batchId`, `details.before` и `details.after`, плюс явный **маркер создания** для строк, которые партия создала.

### Отмена

`POST /membership/batches/:id/undo` (разрешение: создатель партии или `Permissions.server.admin`). Это отказывает если партия не `completed` или старше чем **30-дневное окно отмены**. `BatchUndoHelper.undo()` затем:

1. Загружает строки аудита партии и **группирует их по `(module, entityType, entityId)`.** Сущность, затронутая несколько раз внутри одной партии, обращена **один раз**, обратно в её истинное до-партии состояние — самый ранний before-image, или удаление если партия её создала. Вот почему отмена не наивно повторяет каждую строку: восстановление промежуточного mid-партии снимка было бы неправильно.
2. Для каждой сущности, запускает **защиту конфликта сначала**: `auditLog.hasLaterModification()` спрашивает если ли *более позднее* аудит-вхождение для той же `(module, entityType, entityId)` вне этой партии. Если так, сущность была отредактирована после импорта — она **пропущена и о ней сообщено**, никогда не перезаписана. Это повторно использует сам журнал аудита как детектор модификации; нет столбцов `modifiedAt` на каких-либо таблицах.
3. Обращает per записанный op, разрешая `{ dbModule, table }` из реестра и используя обобщенные записи Kysely:
   - **created** → hard-удалить строку.
   - **updated** → написать `details.before` обратно.
   - **deleted** → пересоздать `details.before` (update-or-insert если строка с этим id переплыла).
4. Каждое обращение само аудируется (`action: "<entityType>_undone"`, нет `batchId` — отмена-отмены вне области).

Op выбран из явного **маркера создания**, не выведен из отсутствия before-image — легитимный пустой before-image или обрезанная строка не должны быть приняты за создание.

Результирующий полезный нагруженный — `{ restored, skippedConflicts: [...], failed: [...], status }`; партия переходит в `undone` (clean) или `partial`. **Есть нет кросс-DB транзакции** — отмена best-effort per строка, то же ограничение, которое Planning Center документирует для объединенных профилей.

:::warning Сущности с побочными эффектами нуждаются в hook `onUndo`
Обращение создания `groupMember` также должно написать `groupMemberHistory` ("left"), или аналитика чесания молча разбивается — стоящий инвариант рабочего пространства. Такие сущности регистрируют callback `onUndo` в `AUDIT_REGISTRY`, который возвращает `true` когда он полностью обработал обращение, обходя обобщенный путь. `groupMembers` — канонический случай (ключ по row id на явном пути но по `personId` на массовых конечных точках, и history-отслеживается на каждом add/remove).
:::

## Потребительские поверхности

Обе admin-поверхности **в работе**; намерение:

| Поверхность | Репо | Цель |
|---------|------|---------|
| **Страница журнала аудита** | B1Admin (ManageChurch → Audit Log) | Фильтр по модулю/категории/пользователю/сущности и отрисовать old→new diffs — для редактирования путем диффирования против предыдущего вхождения аудита сущности, для удаления из `details.before`. Поддержкой `GET /membership/auditlogs`, гейтом по `Permissions.server.admin`. |
| **Страница партий** | B1Admin (тот же hub Settings) | Список партий со статусом и количеством, **View Results** (строки аудита партии via `GET /membership/batches/:id/results`), и кнопка **Undo**, которая поверхности отчет пропущенного-конфликта / отказа. |
| **Импорт партий** | B1Transfer | Открыть партию, отправить `X-Batch-Id` на её нормальные вызовы сохранения, завершить в конце — импорты становятся отменяемы без новых конечных точек импорта. Наследие `importKey` остается как creates-только маркер лдоговора, замещен для отмены. |

## Ловушки будущих изменений должны не регрессировать

- **Вставки аудита должны оставаться ожидаемыми.** Un-awaited `AuditLogHelper.log(...)` опускается Lambda-замораживанием. Собрать обещания и `await Promise.allSettled` перед возвратом.
- **Kysely опускает `undefined` из `.set()`/`.values()`.** На восстановлении, очищенный столбец выживет неприкосновенным. `BatchUndoHelper` преобразует каждое отсутствующее поле в явный `null` (`nullify`) — никогда не обходите его для "более быстрой" прямой записи.
- **Сохранение должно оставаться далеко выше окна отмены.** `AuditLogRepo.deleteOld()` запускается на ночном таймере (по умолчанию 365-дневное сохранение); окно отмены 30 дней. Если сохранение когда-либо снизится к окну, реестры отмены получат очищение из-под открытых партий.
- **Обрезанные строки не отменяемы.** `{ truncated: true }` полезный нагруженный не имеет before/after образа; отмена сообщает его как `failed`, никогда не угадывает.
- **Упорядочение модуля-запись-затем-аудит.** Никогда не перемещайте вставку аудита впереди реальной записи и держите её strict-in-batch / advisory-in-normal.
